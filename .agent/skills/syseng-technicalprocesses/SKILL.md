# Systems Engineering Technical Processes

This skill codifies the technical processes from the INCOSE Systems Engineering Handbook (v5), offering structured guidance for managing the life cycle of complex systems. It should be leveraged when systems thinking, rigorous life-cycle oversight, multi-stakeholder interactions, or well-defined inputs/outputs (receivables/deliverables) between roles are required.

## Key Systems Engineering Principles

1. **Holistic Focus**:
   - Systems Engineering focuses on the whole, examining a problem in its entirety rather than just its individual parts.

2. **Boundaries, Interfaces, and Hierarchies**:
   - Systems are defined by their boundaries and how they interface with external entities. They are often composed of hierarchical architectures.

3. **Form Follows Function Follows Purpose**:
   - The physical structure or **form** of a system is dictated by its required **functions**.
   - These functions, in turn, are derived directly from the ultimate **purpose**, mission, or stakeholder needs the system is built to address.

4. **Life Cycle Approach**:
   - Every system has a life cycle from concept and development to operations, maintenance, and eventual disposal.

5. **Balanced Solutions**:
   - A balanced system solution requires trade-offs. Design decisions must consider opposing constraints (e.g., cost, performance, schedule, risk).

6. **Managed Risks and Opportunities**:
   - All systems have inherent risks and opportunities that must be proactively managed throughout the life cycle.

7. **The V-Model**:
   - A graphical representation of the systems development life cycle.
   - **Left Side (Decomposition & Definition)**: Focuses on defining needs, requirements, architecture, and design. Drives from the system level down to detailed elements (Validation).
   - **Bottom (Realization)**: Implementation of the lowest-level system elements.
   - **Right Side (Integration & Testing)**: Focuses on assembling elements and verifying they meet requirements, moving from detailed elements up to the complete system (Verification).

8. **Verification and Validation (V&V)**:
   - **Validation**: "Are we building the *right* system?" (Ensuring the system addresses stakeholder needs and real-world expectations).
   - **Verification**: "Are we building the system *right*?" (Ensuring the system complies with its specified requirements and architecture).

## The 14 Technical Processes (Structured by V-Model Phases)

### Phase 1: Concept & Definition (Left Side of V-Model)

#### 1. Business or Mission Analysis (BMA)
- **Summary**: Defines the strategic problem/opportunity, characterizes the solution space, and identifies potential solution classes.
- **Key Inputs**: Organization strategic plan, Concept of Operations (ConOps), Life cycle concepts, Constraints.
- **Key Outputs**: BMA strategy/approach, Problem/opportunity statement, Alternative solution classes, Critical performance measurement needs.

#### 2. Stakeholder Needs and Requirements Definition (SNRD)
- **Summary**: Transforms operational concepts into a defined set of stakeholder needs and subsequently into formal stakeholder requirements.
- **Key Inputs**: ConOps, Constraints, Problem/opportunity statement, Alternative solution classes.
- **Key Outputs**: Validated stakeholder needs and requirements, Stakeholder identification, Validation criteria, Traceability mapping.

#### 3. System Requirements Definition (SRD)
- **Summary**: Transforms stakeholder expectations into technical system requirements that guide system design without implementation bias.
- **Key Inputs**: Stakeholder needs and requirements, System viewpoints/models, Constraints.
- **Key Outputs**: System requirements, Verification criteria, System requirements definition report/records.

#### 4. System Architecture Definition (SAD)
- **Summary**: Generates system architecture alternatives, evaluates them, and selects frameworks to conceptualize the system and trace requirements down to system elements.
- **Key Inputs**: System requirements, Constraints, Alternative solution classes.
- **Key Outputs**: System architecture description, System architecture rationale, System interface definition, Traceability mapping.

#### 5. Design Definition (DD)
- **Summary**: Provides sufficient detailed data and information about the system and its elements to enable implementation, ensuring it aligns with the architecture.
- **Key Inputs**: System architecture description, System interface definition, System requirements.
- **Key Outputs**: System design description, System design rationale, System design characteristics, System element description.

#### 6. System Analysis (SA)
*Note: A cross-cutting process utilized throughout the V-Model.*
- **Summary**: Performs rigorous technical assessments to provide quantitative basis for decision-making regarding requirements, architecture, design, and verification.
- **Key Inputs**: System analysis requests, Analysis situations, Life cycle concepts, Constraints.
- **Key Outputs**: System analysis strategy/approach, System analysis report, Traceability mapping.

---

### Phase 2: Realization (Bottom of V-Model)

#### 7. Implementation (IMPL)
- **Summary**: Realizes a specified system element by developing, procuring, or fabricating it based on the detailed design.
- **Key Inputs**: System architecture description, System design description, System element description, System interface definition.
- **Key Outputs**: System element, System element description, Training materials, Implementation report.

---

### Phase 3: Integration & Testing (Right Side of V-Model)

#### 8. Integration (INT)
- **Summary**: Aggregates implemented system elements to form complete or partial system configurations, ensuring interfaces function appropriately.
- **Key Inputs**: System interface definition, System element, Accepted/Reused system elements, Architecture/Design descriptions.
- **Key Outputs**: Integrated system/element, Integration procedure, Integration report.

#### 9. Verification (VER)
- **Summary**: Provides objective evidence that the realized system or system elements fulfill their specified technical requirements and characteristics.
- **Key Inputs**: Verification criteria, System requirements, Integrated system/element, Integration report.
- **Key Outputs**: Verification procedure, Verified system, Verified system requirements, Verification report.

#### 10. Transition (TRAN)
- **Summary**: Establishes the capability to provide services specified by stakeholder requirements within the target operational environment.
- **Key Inputs**: Verified and Validated system, System interface definition, Training materials, Verification/Validation reports.
- **Key Outputs**: Installed system, Trained personnel, Installation procedure, Transition report.

#### 11. Validation (VAL)
- **Summary**: Provides objective evidence that the system, when deployed, accomplishes its intended use in its operational environment by its intended users.
- **Key Inputs**: Validation criteria, Stakeholder needs and requirements, Installed system, Transition report.
- **Key Outputs**: Validated system, Validation procedure, Validated stakeholder needs and requirements, Validation report.

---

### Phase 4: Utilization & Support (Post V-Model)

#### 12. Operation (OPER)
- **Summary**: Uses the system to deliver its specified services for the benefit of the operating organization.
- **Key Inputs**: Installed system, Trained personnel, Transition report, Maintenance and logistics report.
- **Key Outputs**: Operational system, Operation procedure, Operation report.

#### 13. Maintenance (MAINT)
- **Summary**: Sustains the system's capability to provide services, addressing operations support, logistics, and material management to maintain performance and availability.
- **Key Inputs**: Operational system, Trained personnel, Operation report, Life cycle concepts.
- **Key Outputs**: Maintained and sustained system, Maintenance and logistics procedure, Maintenance and logistics report.

#### 14. Disposal (DISP)
- **Summary**: Removes the system (or elements) from operation, handling the safe and secure deactivation, disassembly, and management of waste/hazardous materials.
- **Key Inputs**: Operational system, Maintained/sustained system, Operation report.
- **Key Outputs**: Disposed system, Disposal procedure, Disposal report.
