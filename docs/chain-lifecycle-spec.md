# chain-lifecycle — Runtime Contract Specification

This document specifies the **chain protocol and runtime contract** for
Event-Driven Digital Twin (EDDT) notifications: the envelope types that every
Snapshot and Delta carries, the lifecycle events that govern chain birth,
death, and reset, the runtime helper functions that `delta-gen`-emitted code
calls into, the consumer state machine that manages gap-tolerant apply, and the
constructor primitives producers use to mint chain anchors.

This specification is **normative**. Conforming runtime, producer, and consumer
implementations SHALL satisfy the requirements in §5. Sections explicitly
marked *(informative)* are non-normative.

A reader unfamiliar with the broader EDDT should first read
[eddt-context.md](eddt-context.md) for the architectural context and
[eddt-glossary.md](eddt-glossary.md) for the canonical vocabulary. The
high-level needs this contract contributes to are listed in
[eddt-needs.md](eddt-needs.md) and traced upward from the needs in §3 below.
The generator that emits the code calling these runtime functions is specified
in [delta-gen-spec.md](delta-gen-spec.md).

- **Status:** Draft
- **Version:** 0.1
- **Date:** 2026-06-03

## 1. Purpose and Scope

### 1.1 Purpose

This specification establishes the runtime-side contract that
`delta-gen`-generated `Apply` / `Diff` / `Coalesce` code calls into, and the
chain-protocol commitments that surround the algebra. A conforming runtime
implementation:

- Provides the envelope types `Header`, `Provenance`, and `FieldDelta[T]` (§5.1).
- Exposes the constructor primitives producers use to mint chain anchors (§5.2).
- Exposes the runtime contract functions invoked by generated code (§5.3).
- Enforces the chain-graph invariants on every notification it accepts (§5.4).
- Governs chain lifecycle, cadence, and ownership (§5.5–§5.7).
- Implements the consumer state machine that preserves exactly-once apply
  semantics over at-least-once transport (§5.8).

The generated code that calls these functions is the substrate on which EDDT
Snapshots are propagated, replicated, persisted, and queried; the algebra it
implements is specified in [delta-gen-spec.md](delta-gen-spec.md).

### 1.2 In Scope

- The `Header` and `Provenance` struct shapes and the `FieldDelta[T]` tri-state
  value.
- Chain lifecycle events: birth, delta emission, cadence re-anchoring,
  terminator emission, producer rotation, and disposal.
- Constructor primitives for chain anchors and terminators.
- Runtime contract functions `HeaderAfterApply` and `HeaderForDiff`.
- The algebraic invariants the runtime enforces.
- The consumer state machine: frontier, taint set, gap-tolerant apply,
  late-arrival policies, and anchor recovery.
- Snapshot cadence policy and the single-writer chain-ownership requirement.

### 1.3 Out of Scope

- Per-entity Snapshot and Delta type definitions — these are authored per
  entity by domain modellers and consumed by `delta-gen`.
- The `delta-gen` code-emission rules and the per-field delta algebra — see
  [delta-gen-spec.md](delta-gen-spec.md).
- Transport mechanics: the subject-naming standard, broker topology, retention
  configuration, and consumer-naming conventions. This specification assumes a
  transport whose subjects follow a sufficient subject-naming standard (§8.8);
  it does not define that standard.
- Tier translation between Bronze and Silver (the observation ↔ fact vocabulary
  seam), multi-source aggregation, and downsampling / retention families. These
  compose on top of this contract but introduce no new runtime obligation.
- Per-entity catalogue policy: chain-reset triggers, snapshot-cadence values,
  and taint-recovery thresholds. This specification defines the mechanism;
  concrete values are domain catalogue elicitation work.

## 2. Notation and Conventions

### 2.1 Conformance Language

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD",

[[RFC2119]](#7-references) and RFC 8174 [[RFC8174]](#7-references) when, and
only when, they appear in all capitals, as shown here.

These conformance keywords bind only `R-CL-NNN` requirement statements (§5).
Need statements (§3) use the INCOSE canonical form (§2.2) and do not carry
RFC 2119 obligation force.

### 2.2 Need Statement Form

Each `N-CL-NNN` need is stated in the INCOSE Guide to Writing Requirements
canonical form [[INCOSE-GtWR]](#7-references):

> *"[Stakeholder class] needs [what] so that [why]."*

Where:

- *[Stakeholder class]* is a noun naming the role or coined class that
  originates the need; see §3.1 for the closed register of roles.
- *[what]* is the capability, quality, or constraint expressed in domain terms.
- *[why]* is a brief rationale; the full rationale appears as the
  `Rationale (A1)` bullet of the entry.

Need statements describe expectations; they do not impose obligations in the
RFC 2119 sense. Requirements derived from a need (§5) carry the obligation
force.

### 2.3 Identifier and Reference Conventions

Identifiers in this specification follow flat namespaces with the domain prefix
`CL`:

- **`N-CL-NNN`** — needs (this document, §3).
- **`S-CL-NN`** — subsystem identifiers (this document, §4).
- **`R-CL-NNN`** — requirements (this document, §5).
- **`E-CL-NNN`** — errata (this document, §6).

Identifiers do not encode hierarchy; sub-section breaks are structural.
Cross-references within this document appear inline as e.g. `R-CL-014` and
resolve via explicit `<a id>` anchors in section headers; within-file
navigation uses the generated TOC.

External references cite the source document by short name (e.g. *delta-gen
spec*) or by formal citation in §7 References. INCOSE attribute references
appear inline as `(A18)`, `(A5)`, etc.; attribute definitions live in the
[INCOSE Guide to Writing Requirements](#7-references).

### 2.4 Code and Type Notation

Go syntactic fragments appear in fenced code blocks tagged `go`. Type
definitions are presented in Go syntax; the runtime contract is language-neutral
in spirit, and a conforming implementation in another language SHALL preserve
the field set and semantics established here. `*T` denotes a nullable/optional
value (`nil` denotes absence); `[]T` denotes an ordered slice; generic type
parameters are written `[T any]` per Go 1.18+.

A small synthesised example — a pump on a manufacturing line, with a single
payload field — illustrates the minimum shape of a Snapshot that embeds the
chain `Header`:

```go
type PumpSnapshot struct {
    runtime.Header
    SerialNumber string  `eddt:"entity.key"`
    PressureKPa  float32
}
```

Where this specification refers to Go's standard library or to the
`go.resystems.io/eddt/runtime` package without further qualification, it refers
to the version current at the document's status date.

## 3. High-Level Needs

### 3.1 Stakeholder Roles

The following roles originate the needs in §3.2. Each role is a bounded
responsibility; `(A18)` on each need lists one or more of these roles.

| Role                   | Responsibility                                                                                                                | Primary lifecycle stage |
|:-----------------------|:------------------------------------------------------------------------------------------------------------------------------|:------------------------|
| **Producer**           | Holds single-writer authority for a chain; mints anchors, emits deltas and cadence Snapshots, and runs reset/close protocols. | Runtime                 |
| **Twin maintainer**    | Holds a local Snapshot and folds incoming deltas via `Apply` to keep it current; maintains per-chain frontier and taint.      | Runtime                 |
| **Analytics consumer** | Reads stored Snapshots and deltas and reconstructs or queries historical state, weighting results by taint signatures.        | Analysis                |
| **Platform operator**  | Operates the deployment: monitors chain health, governs recovery, disposal, retention, and producer failover.                 | Sustainment             |

### 3.2 Aspirational Needs

#### <a id="n-cl-001"></a>`N-CL-001` — Convergent Replication

*A Twin Maintainer needs every twin applying the same ordered delta sequence to
converge on identical Snapshot values, so that distributed replicas stay
consistent without coordinated writes.*

- **Stakeholder(s) (A18):** Twin maintainer
- **Source (A5):** Platform need [N-EDDT-002][n-eddt-002] (Orientation &
  Modelling); engineering convention — deterministic state-machine replication
  under eventual consistency
- **Rationale (A1):** The chain is a per-entity, single-writer, ordered log.
  Deterministic, gap-aware application of that log is the property that lets
  independent consumers materialise the same state from the same chain without a
  consensus protocol; it is the foundation of the EDDT's eventual consistency.
- **Validation:** A Twin Maintainer, observing two replicas seeded from the same
  Snapshot and fed the same ordered delta sequence, can defensibly assert the
  replicas hold equal Snapshot values without inspecting each other's state.
- **Priority (A32):** High
- **Criticality (A33):** Essential

#### <a id="n-cl-002"></a>`N-CL-002` — Gap-Tolerant Consumption

*A Twin Maintainer needs to keep applying notifications and produce usable state
under message loss and reordering, with the missing positions explicitly
tracked, so that a single dropped delta does not stall the twin or silently
corrupt it.*

- **Stakeholder(s) (A18):** Twin maintainer, Analytics consumer
- **Source (A5):** Platform need [N-EDDT-008][n-eddt-008] (Data Completeness &
  Fidelity); [Frontier][frontier] and [Taint][taint] in the glossary
- **Rationale (A1):** Real transports deliver at-least-once and out-of-order.
  Applying across gaps while recording the skipped positions as taint keeps the
  twin live and bounds the uncertainty, rather than blocking on strict
  contiguity or pretending completeness.
- **Validation:** A Twin Maintainer fed a stream with a dropped position can
  defensibly assert it still holds the latest applied state and can report
  exactly which positions are outstanding.
- **Priority (A32):** High
- **Criticality (A33):** Essential

#### <a id="n-cl-003"></a>`N-CL-003` — Lineage and Provenance

*An Analytics Consumer needs every notification to carry append-only lineage
and a record of known source gaps, so that any reconstructed state can be
explained, audited, and trust-weighted from its inputs.*

- **Stakeholder(s) (A18):** Analytics consumer, Platform operator
- **Source (A5):** Platform needs [N-EDDT-008][n-eddt-008] (Data Completeness &
  Fidelity), [N-EDDT-002][n-eddt-002] (Orientation & Modelling); [Data
  Provenance][data-provenance] in the glossary
- **Rationale (A1):** Provenance accumulates across tiers and Apply operations
  as an immutable audit trail; embedded source-taint annotations let downstream
  consumers weight or exclude data of known-partial lineage. Rewriting or
  dropping lineage would destroy the basis for trust assessment.
- **Validation:** An Analytics Consumer can defensibly trace any stored state
  back through the components that contributed to it and identify which source
  positions were known-missing at emission time.
- **Priority (A32):** High
- **Criticality (A33):** Important

#### <a id="n-cl-004"></a>`N-CL-004` — Chain Lifecycle Governance

*A Producer needs an unambiguous protocol for starting, re-anchoring, handing
over, and closing a chain, so that an entity's history has well-defined birth,
continuation, succession, and death without coordinated writers.*

- **Stakeholder(s) (A18):** Producer, Platform operator
- **Source (A5):** Platform needs [N-EDDT-010][n-eddt-010] (System
  Evolvability), [N-EDDT-011][n-eddt-011] (Data Retention & Lifecycle)
- **Rationale (A1):** Entities are created, re-identified, and retired. A fixed
  set of lifecycle events — birth, cadence, terminator, symmetric reset, and
  disposal — gives every chain a finite, navigable shape and lets consumers
  reason about succession and termination deterministically.
- **Validation:** A Producer can defensibly assert that every chain it owns has
  exactly one birth, a strictly-ordered body, and at most one terminator, and
  that successor chains are reachable from their predecessors.
- **Priority (A32):** High
- **Criticality (A33):** Essential

#### <a id="n-cl-005"></a>`N-CL-005` — Bounded Recovery Latency

*A Twin Maintainer needs to recover current chain state in bounded time after
subscribing late or losing local state, so that catch-up cost is governed by
cadence rather than by replaying an unbounded delta history.*

- **Stakeholder(s) (A18):** Twin maintainer, Platform operator
- **Source (A5):** Platform needs [N-EDDT-005][n-eddt-005] (Temporal
  Performance), [N-EDDT-009][n-eddt-009] (Platform Operational Health)
- **Rationale (A1):** Periodic full-state cadence Snapshots are reusable chain
  anchors. A consumer recovers by fetching the latest Snapshot and resuming from
  its Sequence, so recovery latency is bounded by the cadence interval rather
  than by chain length.
- **Validation:** A Twin Maintainer subscribing mid-chain can defensibly assert
  it reaches current state in time bounded by the cadence interval, not by the
  number of deltas since birth.
- **Priority (A32):** Medium
- **Criticality (A33):** Important

#### <a id="n-cl-006"></a>`N-CL-006` — Single-Writer Integrity

*A Producer needs each chain to have at most one active writer at any instant,
so that the Sequence space stays monotonic and free of write conflicts without
distributed coordination.*

- **Stakeholder(s) (A18):** Producer, Platform operator
- **Source (A5):** Platform need [N-EDDT-007][n-eddt-007] (Scalability & Data
  Velocity); [Chain][chain] in the glossary
- **Rationale (A1):** Single-writer ownership is what makes a monotonic Sequence
  achievable without consensus and lets chains shard horizontally across
  producers. Multi-source scenarios are modelled as separate chains merged by an
  aggregator, never as concurrent writers to one chain.
- **Validation:** A Producer can defensibly assert that no two instances emit on
  the same `ChainID` concurrently, and that a failover preserves the monotonic
  Sequence across the handover.
- **Priority (A32):** High
- **Criticality (A33):** Essential

## 4. Architecture

### 4.1 Subsystems

The requirements in §5 are allocated to three subsystems. The split mirrors the
three conformance audiences of this contract.

| ID        | Name            | Counterparty              | Owns                                                                    |
|:----------|:----------------|:--------------------------|:------------------------------------------------------------------------|
| `S-CL-01` | Runtime Library | Generated code; producers | Envelope types, constructors, contract functions, invariants            |
| `S-CL-02` | Producer        | Runtime; consumers        | Anchor minting, cadence, reset/close protocols, single-writer ownership |
| `S-CL-03` | Consumer        | Producers; analytics      | Frontier/taint state, receive protocol, late-arrival, recovery          |

### 4.2 Subsystem Interactions *(informative)*

The Producer (`S-CL-02`) uses the Runtime Library's constructor primitives
(`S-CL-01`, §5.2) to mint anchors and calls `delta-gen`-generated `Diff` (which
in turn calls the Runtime Library's `HeaderForDiff`, §5.3) to emit deltas. The
Consumer (`S-CL-03`) folds those deltas via generated `Apply` (which calls
`HeaderAfterApply`, §5.3), maintaining the per-chain `(Frontier, Taint)` state
of §5.8. The Runtime Library enforces the algebraic invariants of §5.4 inside
the contract functions and constructors, so neither producer nor consumer can
advance a chain through an invalid transition.

## 5. Requirements

### 5.1 Envelope Types

The requirements in this section bind the Runtime Library (`S-CL-01`).

#### <a id="r-cl-001"></a>`R-CL-001` — Header shape

> *The Runtime Library SHALL provide a `Header` type, embedded by every Snapshot
> and Delta, with the following fields and population rules:*
>
> ```go
> type Header struct {
>     EntityID        EntityID     // content-hash of the entity key
>     ChainID         string       // chain identity (opaque)
>     PreviousChainID *string      // set only on the birth Snapshot
>     NextChainID     *string      // set only on the terminator Snapshot
>     Closed          *time.Time   // set only on the terminator Snapshot
>     Sequence        uint64       // 0 = birth; strictly increasing thereafter
>     EffectiveAt     time.Time    // domain time the state/change took effect
>     PublishedAt     time.Time    // wall-clock emission instant
>     Provenance      []Provenance // accumulated lineage across tiers
> }
> ```
>
> *`EntityID`, `ChainID`, `Sequence`, `EffectiveAt`, `PublishedAt`, and
> `Provenance` SHALL be populated on every notification. `EntityID` SHALL be
> equal across all notifications on a chain. `ChainID` SHALL be unique across
> the deployment and stable for the chain's lifetime.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-001` Convergent Replication; `N-CL-004` Chain
  Lifecycle Governance
- **Source (A5):** Observed implementation (`runtime.Header`); chain-lifecycle
  upstream §3.1
- **Rationale (A1):** The Header is the chain envelope: identity (`EntityID`,
  `ChainID`), causal position (`Sequence`), bitemporal anchoring (`EffectiveAt`,
  `PublishedAt`), and lineage (`Provenance`). A uniform `EntityID` content-hash
  lets cross-chain entity equality be a plain comparison (see delta-gen spec
  R-DG-034/R-DG-035).
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P0
- **Criticality (A33):** Critical

#### <a id="r-cl-002"></a>`R-CL-002` — Anchor-only fields

> *The fields `PreviousChainID`, `NextChainID`, and `Closed` are the anchor-only
> fields. A producer SHALL set them with non-`nil` values only as follows, and
> SHALL leave them `nil` on every other notification:*
>
> - *`PreviousChainID` — only on the birth Snapshot (`Sequence == 0`), and only
>   when the chain succeeds a predecessor;*
> - *`NextChainID` — only on the terminator Snapshot, and only when a successor
>   chain exists;*
> - *`Closed` — only on the terminator Snapshot.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-004` Chain Lifecycle Governance
- **Source (A5):** Observed implementation (`runtime.Header`); chain-lifecycle
  upstream §3.1.1
- **Rationale (A1):** The anchor-only fields are chain-protocol metadata that
  describe the chain's birth and death, not the entity. Confining them to the
  anchors keeps intermediate notifications unambiguous and lets consumers detect
  succession and termination by field presence alone.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-01` Runtime Library; `S-CL-02` Producer
- **Priority (A32):** P1
- **Criticality (A33):** High

#### <a id="r-cl-003"></a>`R-CL-003` — ChainID format

> *`ChainID` SHALL be an opaque string. A conforming `ChainID` minting scheme
> SHALL satisfy: (1) uniqueness across producers without coordination;
> (2) stability — immutable for the chain's lifetime; and (3) disposal
> friendliness — it SHALL NOT embed PII or other disposal-sensitive identifiers.
> It SHOULD additionally be amenable to time-ordered clustering. The default
> scheme SHALL be UUIDv7 [[RFC9562]](#7-references); deployments MAY select an
> alternative scheme that meets these properties.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-004` Chain Lifecycle Governance; `N-CL-006`
  Single-Writer Integrity
- **Source (A5):** chain-lifecycle upstream §3.1.2; [[RFC9562]](#7-references)
- **Rationale (A1):** Coordination-free uniqueness lets any producer mint a
  chain identity locally; stability lets it serve as a durable key; disposal
  friendliness keeps per-chain deletion (§5.5) from correlating with PII.
  Time-ordered clustering aids analytical scan performance.
- **V&V method (A2):** Analysis
- **Allocation (A8):** `S-CL-01` Runtime Library; `S-CL-02` Producer
- **Priority (A32):** P2
- **Criticality (A33):** Moderate

#### <a id="r-cl-004"></a>`R-CL-004` — Provenance shape and accumulation

> *The Runtime Library SHALL provide a `Provenance` type carrying `PublishedAt`,
> optional `ValidUntil`, `Solution`, `Component`, `Instance`, optional
> `Metadata`, and optional `Gaps` (`[]SequenceRange`). A producer SHALL append a
> `Provenance` entry naming itself on every notification it emits; an aggregator
> SHALL append its own entry while preserving upstream entries. Provenance SHALL
> be append-only: no entry SHALL be rewritten, reordered, or removed.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-003` Lineage and Provenance
- **Source (A5):** Observed implementation (`runtime.Provenance`,
  `runtime.SequenceRange`); chain-lifecycle upstream §3.2, §3.2.1
- **Rationale (A1):** An append-only lineage chain is the audit substrate that
  makes reconstructed state explainable and reproducible. Preserving upstream
  entries through aggregation keeps the full contributing history intact.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-01` Runtime Library; `S-CL-02` Producer
- **Priority (A32):** P1
- **Criticality (A33):** High

#### <a id="r-cl-005"></a>`R-CL-005` — Source-taint annotation immutability

> *When a producer cites an upstream source chain, it MAY populate the `Gaps`
> field of the corresponding `Provenance` entry with a snapshot-copy of its
> then-current taint set for that source. Once emitted, `Gaps` SHALL NOT be
> revised by subsequent late-arrival processing; already-emitted notifications
> retain their original `Gaps` even if upstream taint later lifts.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-003` Lineage and Provenance
- **Source (A5):** chain-lifecycle upstream §3.2.2
- **Rationale (A1):** `Gaps` is an immutable lineage annotation describing what
  was known-missing at emission time. Freezing it preserves the historical
  trust signal; revising it retroactively would rewrite the audit record.
- **V&V method (A2):** Analysis
- **Allocation (A8):** `S-CL-02` Producer
- **Priority (A32):** P2
- **Criticality (A33):** Moderate

#### <a id="r-cl-006"></a>`R-CL-006` — FieldDelta tri-state

> *The Runtime Library SHALL provide `FieldDelta[T any]` carrying an `Op` of
> type `FieldDeltaOp` and a `Value T`, together with the constants `OpIgnore`
> (0, leave the field unchanged — the zero value), `OpAssert` (1, set the field
> to `Value`), and `OpRetract` (2, reset the field to its zero value). These are
> the carrier and operations for `delta.clearable` fields; the per-field
> application is generator-emitted (see §8.7 and delta-gen spec R-DG-016).*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-004` Chain Lifecycle Governance
- **Source (A5):** Observed implementation (`runtime.FieldDelta`,
  `runtime.FieldDeltaOp`); chain-lifecycle upstream §3.3; reconciled per
  `E-CL-003`
- **Rationale (A1):** A single tri-state carrier expresses ignore / assert /
  retract for optional fields. Making `OpIgnore` the zero value means a
  zero-valued `FieldDelta` is an explicit no-op, which composes cleanly with the
  generated per-field switch.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P1
- **Criticality (A33):** High

#### <a id="r-cl-007"></a>`R-CL-007` — Snapshot embeds Header

> *A conforming Snapshot type and its companion Delta type SHALL each embed
> exactly one `Header` field, recognised by type rather than by name. A
> `delta.nested` companion type SHALL NOT embed `Header`.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-001` Convergent Replication
- **Source (A5):** chain-lifecycle upstream §3.4; delta-gen spec R-DG-002,
  R-DG-008
- **Rationale (A1):** The constructor primitives and contract functions operate
  on the embedded Header by type identity; a single, unambiguous envelope per
  Snapshot is what lets generic runtime code populate and validate it.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P1
- **Criticality (A33):** High

### 5.2 Constructor Primitives

The requirements in this section bind the Runtime Library (`S-CL-01`); producers
(`S-CL-02`) consume them.

#### <a id="r-cl-008"></a>`R-CL-008` — Birth Snapshot constructor

> *The Runtime Library SHALL expose a constructor that mints a birth Snapshot
> for an entity, populating the embedded Header with `Sequence = 0`,
> `Closed = nil`, `NextChainID = nil`, the supplied `EntityID`, freshly-minted
> `ChainID`, optional `PreviousChainID`, `EffectiveAt`, `PublishedAt`, and
> `Provenance`. The supplied `ChainID` SHALL be a freshly-minted unique value
> (R-CL-003) and the `Provenance` SHOULD name the producer.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-004` Chain Lifecycle Governance
- **Source (A5):** chain-lifecycle upstream §5.1
- **Rationale (A1):** Minting birth Snapshots through a single primitive
  guarantees the `Sequence == 0` / anchor-field discipline (R-CL-002) is applied
  consistently and that hand-rolled Header construction cannot drift from the
  contract.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P1
- **Criticality (A33):** High

#### <a id="r-cl-009"></a>`R-CL-009` — Cadence Snapshot constructor

> *The Runtime Library SHALL expose a constructor that mints a cadence (non-
> terminator, `Sequence > 0`) full-state Snapshot from the prior chain state,
> carrying `EntityID` and `ChainID` from the prior Header, `Closed = nil`,
> `NextChainID = nil`, the supplied `Sequence`, `EffectiveAt`, `PublishedAt`, and
> `Provenance` accumulated as `prior.Provenance ⊕ provenance`. It SHALL enforce
> the constructor preconditions of R-CL-011.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-004` Chain Lifecycle Governance; `N-CL-005`
  Bounded Recovery Latency
- **Source (A5):** chain-lifecycle upstream §5.2
- **Rationale (A1):** Cadence Snapshots are the reusable anchors that bound
  recovery latency (§5.6, §5.8); minting them through a primitive enforces the
  monotonic-Sequence and provenance-accumulation invariants by construction.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P1
- **Criticality (A33):** High

#### <a id="r-cl-010"></a>`R-CL-010` — Terminator Snapshot constructor

> *The Runtime Library SHALL expose a constructor that mints a terminator
> Snapshot from the prior chain state, setting `Closed` to a non-`nil` instant,
> optional `NextChainID`, `EntityID` and `ChainID` from the prior Header, the
> supplied `Sequence`, `EffectiveAt`, `PublishedAt`, and accumulated
> `Provenance`. After it returns, the runtime SHALL NOT permit further
> notifications on that `ChainID` (R-CL-020). It SHALL enforce the preconditions
> of R-CL-011.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-004` Chain Lifecycle Governance
- **Source (A5):** chain-lifecycle upstream §5.3
- **Rationale (A1):** A single terminator primitive guarantees `Closed` is set,
  the Sequence advances past every prior position, and chain finiteness
  (R-CL-020) takes effect atomically with the close.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P1
- **Criticality (A33):** High

#### <a id="r-cl-011"></a>`R-CL-011` — Constructor validation

> *Each constructor SHALL validate its preconditions and return an error on
> violation; a conforming runtime MUST NOT silently emit a notification with a
> known invariant violation. For the cadence and terminator constructors the
> preconditions SHALL be: `sequence > prior.Sequence` (R-CL-015),
> `effectiveAt >= prior.EffectiveAt` (R-CL-017), and the prior chain SHALL NOT
> have been terminated (R-CL-020).*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-004` Chain Lifecycle Governance
- **Source (A5):** chain-lifecycle upstream §5.4
- **Rationale (A1):** Validating at mint time surfaces invariant violations
  before a malformed notification can enter the chain, where it would corrupt
  every downstream consumer's reconstruction.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P0
- **Criticality (A33):** High

### 5.3 Runtime Contract Functions

The requirements in this section bind the Runtime Library (`S-CL-01`); generated
code calls these functions.

#### <a id="r-cl-012"></a>`R-CL-012` — HeaderAfterApply

> *The Runtime Library SHALL expose `HeaderAfterApply(s, d Header) (Header,
> error)`, producing the result Header for `Apply(s, d)` with `EntityID`/
> `ChainID` from `s`, `PreviousChainID = NextChainID = Closed = nil`, `Sequence`
> / `EffectiveAt` / `PublishedAt` from `d`, and `Provenance = s.Provenance ⊕
> d.Provenance`. It SHALL return a non-nil error, leaving state unchanged, when
> any of the following holds: `s.ChainID != d.ChainID` (R-CL-014);
> `s.EntityID != d.EntityID`; either `EntityID` is the zero value (E-CL-002);
> `d.Sequence <= s.Sequence` (R-CL-015, R-CL-016); `d.EffectiveAt <
> s.EffectiveAt` (R-CL-017); or `s.Closed != nil` (R-CL-020).*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-001` Convergent Replication; `N-CL-003` Lineage
  and Provenance
- **Source (A5):** Observed implementation (`runtime.HeaderAfterApply`);
  chain-lifecycle upstream §6.1; reconciled per `E-CL-001`, `E-CL-002`
- **Rationale (A1):** `Apply`'s result is a mid-chain notification, so it clears
  the anchor-only fields, advances the causal position from the delta, and
  concatenates provenance. The preconditions are the chain-integrity gate that
  keeps an invalid fold from corrupting state; they are shared with delta-gen
  R-DG-029/R-DG-031 (see §8.7).
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P0
- **Criticality (A33):** Critical

#### <a id="r-cl-013"></a>`R-CL-013` — HeaderForDiff

> *The Runtime Library SHALL expose `HeaderForDiff(a, b Header) (Header, error)`,
> producing the Header for the delta `d` such that `Apply(a, d) == b`, with
> `EntityID`/`ChainID`/`Sequence`/`EffectiveAt`/`PublishedAt` from `b`, cleared
> anchor-only fields, and empty `Provenance` by default (callers supply lineage
> after the call). It SHALL return a non-nil error, leaving state unchanged,
> when `a.ChainID != b.ChainID` (R-CL-014), `a.EntityID != b.EntityID`, either
> `EntityID` is zero (E-CL-002), `b.Sequence < a.Sequence` (R-CL-015), or
> `b.EffectiveAt < a.EffectiveAt` (R-CL-017).*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-001` Convergent Replication
- **Source (A5):** Observed implementation (`runtime.HeaderForDiff`);
  chain-lifecycle upstream §6.2; reconciled per `E-CL-001`, `E-CL-002`
- **Rationale (A1):** `Diff` is a pure transformation that carries no lineage by
  default; coalescing aggregators and audit pipelines attach provenance at the
  call site rather than encoding a global policy in the function.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P0
- **Criticality (A33):** Critical

### 5.4 Algebraic Invariants

The requirements in this section bind the Runtime Library (`S-CL-01`), enforced
through the contract functions (§5.3) and constructors (§5.2). They are
universal claims over the input space and therefore carry `A2 = Analysis`. The
numbering is flat `R-CL`; the parenthetical `Inv. N` notes the upstream
invariant each derives from.

#### <a id="r-cl-014"></a>`R-CL-014` — Chain integrity

> *`Apply` and `Diff` SHALL be defined only when the two operands share a
> `ChainID` and an `EntityID`. Cross-chain or cross-entity application SHALL be
> rejected with a non-nil error.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-001` Convergent Replication
- **Source (A5):** chain-lifecycle upstream §7.4 (`Inv. 4`); delta-gen spec
  R-DG-029, R-DG-030
- **Rationale (A1):** A chain is a single entity's history; folding a delta from
  another chain or entity is meaningless and would silently corrupt state.
- **V&V method (A2):** Analysis
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P0
- **Criticality (A33):** Critical

#### <a id="r-cl-015"></a>`R-CL-015` — Sequence monotonicity

> *Within a chain, `Sequence` SHALL be strictly increasing across successive
> notifications: the birth Snapshot has `Sequence == 0` and every subsequent
> emission has a strictly greater `Sequence`.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-001` Convergent Replication; `N-CL-006`
  Single-Writer Integrity
- **Source (A5):** chain-lifecycle upstream §7.5 (`Inv. 5`); delta-gen spec
  R-DG-029, R-DG-030
- **Rationale (A1):** A strictly monotonic Sequence is the chain's causal order;
  it underpins idempotent delivery, gap detection, and subscription resume.
- **V&V method (A2):** Analysis
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P0
- **Criticality (A33):** Critical

#### <a id="r-cl-016"></a>`R-CL-016` — Gap-tolerant apply

> *`HeaderAfterApply(s, d)` SHALL require `d.Sequence > s.Sequence` — strict
> monotonic progression, not strict contiguity (`d.Sequence == s.Sequence + 1`).
> Skipped Sequences in the open interval `(s.Sequence, d.Sequence)` SHALL be
> recorded as taint by the consumer state machine (§5.8); the Apply algebra
> itself SHALL NOT reject the gap.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-002` Gap-Tolerant Consumption
- **Source (A5):** chain-lifecycle upstream §7.6 (`Inv. 6`)
- **Rationale (A1):** Permitting application across gaps keeps the twin live
  under loss; separating "apply" from "completeness" lets taint carry the
  uncertainty rather than blocking progress.
- **V&V method (A2):** Analysis
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P0
- **Criticality (A33):** Essential

#### <a id="r-cl-017"></a>`R-CL-017` — EffectiveAt non-decrease

> *For any `HeaderAfterApply(s, d)`, the runtime SHALL enforce `d.EffectiveAt >=
> s.EffectiveAt`. Strict monotonicity is not required — simultaneous changes MAY
> share an instant — but `EffectiveAt` SHALL NOT roll backwards within a chain.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-001` Convergent Replication
- **Source (A5):** chain-lifecycle upstream §7.7 (`Inv. 7`); delta-gen spec
  R-DG-029, R-DG-030
- **Rationale (A1):** Domain time must not regress within a single entity's
  history; allowing ties accommodates batched simultaneous changes without
  permitting causal inversion.
- **V&V method (A2):** Analysis
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P1
- **Criticality (A33):** High

#### <a id="r-cl-018"></a>`R-CL-018` — Provenance accumulation

> *`HeaderAfterApply(s, d).Provenance` SHALL equal `s.Provenance ⊕ d.Provenance`
> (slice concatenation). No `Provenance` entry SHALL be dropped, reordered, or
> rewritten.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-003` Lineage and Provenance
- **Source (A5):** chain-lifecycle upstream §7.8 (`Inv. 8`); delta-gen spec
  R-DG-032
- **Rationale (A1):** Apply is the point at which lineage accumulates; making it
  pure concatenation guarantees the audit trail is complete and order-preserving.
- **V&V method (A2):** Analysis
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P1
- **Criticality (A33):** High

#### <a id="r-cl-019"></a>`R-CL-019` — Taint well-definedness

> *For any reconstructed state `s` produced by a sequence of Apply operations
> from a chain's birth Snapshot, the taint set `T(s) = { n : 0 < n < s.Sequence,
> n not applied }` SHALL be computable from the consumer's history of Apply
> operations. Maintenance of `T` is the consumer state machine's responsibility
> (§5.8).*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-002` Gap-Tolerant Consumption
- **Source (A5):** chain-lifecycle upstream §7.11 (`Inv. 11`)
- **Rationale (A1):** Taint is only meaningful if it is exactly computable; a
  well-defined taint set is what lets a consumer report completeness and trigger
  recovery deterministically.
- **V&V method (A2):** Analysis
- **Allocation (A8):** `S-CL-01` Runtime Library; `S-CL-03` Consumer
- **Priority (A32):** P1
- **Criticality (A33):** High

#### <a id="r-cl-020"></a>`R-CL-020` — Chain finiteness

> *If a terminator Snapshot at Sequence `N` has been emitted on a chain, no
> notification with Sequence `> N` is valid on that chain. A conforming runtime
> SHALL reject any attempt to emit, accept, or apply such a notification. A chain
> without an emitted terminator remains open in the strict sense even when its
> producer is quiescent (silence-as-close, R-CL-025, is a fallback outside this
> invariant).*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-004` Chain Lifecycle Governance
- **Source (A5):** chain-lifecycle upstream §7.12 (`Inv. 12`)
- **Rationale (A1):** A terminator is a hard upper bound on the Sequence space;
  enforcing it lets consumers free per-chain state and treat the chain as a
  closed, immutable historical record.
- **V&V method (A2):** Analysis
- **Allocation (A8):** `S-CL-01` Runtime Library
- **Priority (A32):** P1
- **Criticality (A33):** High

### 5.5 Chain Lifecycle Events

The requirements in this section bind the Producer (`S-CL-02`).

#### <a id="r-cl-021"></a>`R-CL-021` — Chain birth

> *A producer SHALL begin each chain with exactly one birth Snapshot minted via
> the birth constructor (R-CL-008): `Sequence == 0`, `Closed == nil`,
> `NextChainID == nil`, a freshly-minted `ChainID`, and a populated `Provenance`
> naming the producer. `PreviousChainID` SHALL be set if and only if the chain
> succeeds a predecessor (R-CL-022).*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-004` Chain Lifecycle Governance
- **Source (A5):** chain-lifecycle upstream §4.2
- **Rationale (A1):** Exactly one birth per chain gives every entity history a
  single, unambiguous origin from which reconstruction proceeds.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-02` Producer
- **Priority (A32):** P1
- **Criticality (A33):** High

#### <a id="r-cl-022"></a>`R-CL-022` — Symmetric reset

> *A producer that hands a chain over to a successor SHALL perform a symmetric
> reset, minting in coordination: a terminator on the predecessor with `Closed
> != nil`, `NextChainID` set to the successor's `ChainID`, and `Sequence`
> strictly greater than every prior Sequence; and a birth on the successor with
> `Sequence == 0`, `PreviousChainID` set to the predecessor's `ChainID`, and a
> freshly-minted distinct `ChainID`. After the reset, no notification with
> `Sequence > terminator.Sequence` SHALL be emitted on the predecessor (R-CL-020).*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-004` Chain Lifecycle Governance
- **Source (A5):** chain-lifecycle upstream §4.3
- **Rationale (A1):** Pairing terminator and birth at handover makes succession
  navigable in both directions (`NextChainID` / `PreviousChainID`) and keeps the
  predecessor a closed, addressable historical record.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-02` Producer
- **Priority (A32):** P2
- **Criticality (A33):** Moderate

#### <a id="r-cl-023"></a>`R-CL-023` — Close without successor

> *A producer that ceases tracking an entity SHALL emit a terminator Snapshot
> with `Closed != nil`, `NextChainID == nil`, and `Sequence` strictly greater
> than every prior Sequence. A consumer receiving such a terminator SHALL free
> per-chain waiting state and SHALL NOT wait for further notifications on that
> `ChainID`.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-004` Chain Lifecycle Governance
- **Source (A5):** chain-lifecycle upstream §4.4
- **Rationale (A1):** A successor-less terminator is the clean end-of-life
  signal that lets consumers reclaim resources deterministically.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-02` Producer; `S-CL-03` Consumer
- **Priority (A32):** P2
- **Criticality (A33):** Moderate

#### <a id="r-cl-024"></a>`R-CL-024` — Producer rotation

> *Producer rotation (failover) transfers single-writer authority for a chain
> between producer instances. It SHALL preserve `ChainID` and the Sequence space
> (the successor's first Sequence SHALL be greater than the predecessor's last
> emitted Sequence), SHALL NOT emit a terminator, SHALL NOT set `PreviousChainID`
> or `NextChainID`, and SHALL be opaque to consumers.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-006` Single-Writer Integrity
- **Source (A5):** chain-lifecycle upstream §4.5
- **Rationale (A1):** Rotation is a change of writer, not a change of chain;
  distinguishing it from chain close (no terminator, no linkage) lets a chain
  survive producer failure without consumers observing a discontinuity.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-02` Producer
- **Priority (A32):** P2
- **Criticality (A33):** Moderate

#### <a id="r-cl-025"></a>`R-CL-025` — Silence-as-close fallback

> *Where a producer fails to emit a terminator, a consumer MAY treat a chain as
> closed after a per-entity quiescence threshold elapses without a new
> notification. Authoritative in-band termination via `Closed != nil` SHALL take
> precedence whenever present. The quiescence threshold is catalogue policy and
> is not specified here.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-004` Chain Lifecycle Governance
- **Source (A5):** chain-lifecycle upstream §4.6
- **Rationale (A1):** Producers can crash mid-life; a fallback close bounds the
  resources a consumer commits to a chain that will never be terminated in-band,
  without overriding an explicit terminator when one does arrive.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-03` Consumer
- **Priority (A32):** P3
- **Criticality (A33):** Optional

#### <a id="r-cl-026"></a>`R-CL-026` — Disposal

> *A conforming runtime SHALL support per-chain disposal by removing every
> storage object keyed by `ChainID`. Field-level redaction SHALL be achievable
> via an `OpRetract` `FieldDelta` (R-CL-006) on the sensitive fields. A runtime
> MAY maintain a natural-key-to-chain index to support cross-chain disposal and
> MAY complement logical retraction with out-of-band rewrites where physical
> irrecoverability is legally required.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-004` Chain Lifecycle Governance
- **Source (A5):** chain-lifecycle upstream §4.7; platform need
  [N-EDDT-011][n-eddt-011] (Data Retention & Lifecycle)
- **Rationale (A1):** `ChainID` opacity (R-CL-003) makes per-chain deletion
  PII-safe; `OpRetract` gives in-band logical erasure for the common case while
  leaving physical rewrite available for regulatory erasure.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-01` Runtime Library; `S-CL-02` Producer
- **Priority (A32):** P2
- **Criticality (A33):** Moderate

### 5.6 Snapshot Cadence

#### <a id="r-cl-027"></a>`R-CL-027` — Hybrid cadence

> *A producer SHALL emit periodic cadence Snapshots on every chain following a
> hybrid pattern: a time-based floor (no chain SHALL run longer than a per-entity

- **Type (A38):** Functional

- **Source (A5):** chain-lifecycle upstream §9
- **Rationale (A1):** The time floor bounds recovery latency predictably; the
  change-driven trigger captures domain-relevant transitions regardless of
  clock. Each covers the other's blind spot.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-02` Producer
- **Priority (A32):** P2
- **Criticality (A33):** Important

### 5.7 Chain Ownership

The requirements in this section bind the Producer (`S-CL-02`).

#### <a id="r-cl-028"></a>`R-CL-028` — Single-writer invariant

> *A conforming runtime SHALL enforce that each `ChainID` has at most one active
> producer at any instant. Concurrent writers to the same chain are prohibited.
> Multi-source scenarios SHALL be represented by separate chains per source,
> merged downstream by an aggregator.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-006` Single-Writer Integrity

- **Rationale (A1):** Single-writer ownership is what makes a monotonic Sequence
  achievable without consensus; modelling multiple sources as separate merged
  chains preserves it.
- **V&V method (A2):** Analysis
- **Allocation (A8):** `S-CL-02` Producer
- **Priority (A32):** P1
- **Criticality (A33):** Essential

#### <a id="r-cl-029"></a>`R-CL-029` — Ownership mapping mechanism

> *A conforming deployment SHALL select and document one chain-to-producer
> mapping mechanism (static configuration, hash-based routing, or registry-based

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-006` Single-Writer Integrity
- **Source (A5):** chain-lifecycle upstream §10.2
- **Rationale (A1):** The mapping mechanism is a deployment choice, but whichever
  is chosen must not admit two writers or a Sequence regression during handover.
- **V&V method (A2):** Analysis
- **Allocation (A8):** `S-CL-02` Producer
- **Priority (A32):** P3
- **Criticality (A33):** Moderate

#### <a id="r-cl-030"></a>`R-CL-030` — Failover continuity

> *A deployment's failover protocol SHALL determine the successor producer,
> establish the predecessor's last emitted Sequence (anchor recovery, §5.8, is a
> sufficient mechanism), and prevent split-brain via fencing tokens, leases, or
> equivalent. Failover SHALL preserve chain identity (R-CL-024): no terminator is

- **Type (A38):** Functional

- **Rationale (A1):** Continuity across writer failure requires answering who

- **Allocation (A8):** `S-CL-02` Producer
- **Priority (A32):** P2
- **Criticality (A33):** Important

### 5.8 Consumer State Machine

The requirements in this section bind the Consumer (`S-CL-03`).

#### <a id="r-cl-031"></a>`R-CL-031` — Per-chain state

> *A conforming consumer SHALL maintain, per subscribed chain, a frontier `F`
> (the highest applied `Sequence`, initially `0` after the birth Snapshot) and a
> taint set `T` (the Sequences strictly between `0` and `F` not yet applied). A
> Sequence enters `T` on a forward gap and leaves `T` when the late-arriving
> notification is applied under the active late-arrival policy (R-CL-033).*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-002` Gap-Tolerant Consumption
- **Source (A5):** chain-lifecycle upstream §8.1
- **Rationale (A1):** Frontier and taint are the complete per-chain bookkeeping
  needed for idempotent delivery, gap detection, and subscription resume.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-03` Consumer
- **Priority (A32):** P1
- **Criticality (A33):** Essential

#### <a id="r-cl-032"></a>`R-CL-032` — Receive protocol

> *On receipt of a notification at Sequence `k` on a chain with state `(F, T)`, a
> conforming consumer SHALL act as follows, giving exactly-once apply semantics
> over at-least-once transport:*
>
> | Condition                            | Action                                                              |
> |:-------------------------------------|:--------------------------------------------------------------------|
> | `k <= F` and `k ∉ T`                 | Already-applied duplicate. Discard.                                 |
> | `k ∈ T`                              | Late arrival. Apply per the active late-arrival policy (R-CL-033).  |
> | `k == F + 1`                         | In-order. Apply; set `F := k`.                                      |
> | `k > F + 1`                          | Forward gap. Apply (R-CL-016); add `{F+1 … k-1}` to `T`; set `F := k`. |
> | Snapshot with `Closed != nil`        | Terminator. Apply; chain closed; MAY free per-chain state.          |
>
> *A consumer SHALL NOT conflate the three concerns of duplicate handling, gap
> handling, and accumulated-loss (taint) tracking.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-001` Convergent Replication; `N-CL-002`
  Gap-Tolerant Consumption
- **Source (A5):** chain-lifecycle upstream §8.2, §8.4
- **Rationale (A1):** The frontier dispatches duplicates trivially, the
  gap-tolerant rule dispatches gaps by applying across them and recording taint,
  and the terminator frees state — three orthogonal concerns handled by one
  deterministic protocol.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-03` Consumer
- **Priority (A32):** P0
- **Criticality (A33):** Essential

#### <a id="r-cl-033"></a>`R-CL-033` — Late-arrival policy

> *A conforming consumer SHALL implement strong rejection as the default
> late-arrival policy: a notification whose Sequence is in `T` is discarded and
> the Sequence remains tainted until cleared by anchor recovery (R-CL-035). A
> consumer MAY additionally implement jitter-buffered apply (holding
> notifications a bounded window to reorder slightly-out-of-order arrivals
> contiguously). For `delta.commutative` fields a future iteration MAY honour
> CRDT-style late-arrival lift; a v1 consumer SHALL accept the tag and fall back
> to strong rejection.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-002` Gap-Tolerant Consumption
- **Source (A5):** chain-lifecycle upstream §8.3
- **Rationale (A1):** Strong rejection is the simplest correct policy and the
  required floor; jitter-buffering and CRDT lift are opt-in optimisations that
  trade latency or complexity for fewer recovery events. See [CRDT][crdt],
  [LWW-Register][last-writer-wins-register-lww-register], and [ORSet][orset].
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-03` Consumer
- **Priority (A32):** P1
- **Criticality (A33):** High

#### <a id="r-cl-034"></a>`R-CL-034` — Anchor recovery

> *A consumer subscribing mid-chain, or whose local state has drifted beyond
> repair, SHALL recover by requesting the latest Snapshot for the chain and, on
> receipt of Snapshot `s`, initialising `F := s.Sequence` and `T := ∅`. If
> `s.Closed != nil` the chain has terminated (the consumer applies `s`, frees
> state, and MAY follow `s.NextChainID` to the successor); otherwise it resumes
> the live stream from `s.Sequence + 1`. First-time subscription, drift
> recovery, and discovery of an already-closed chain SHALL be the same operation.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-005` Bounded Recovery Latency
- **Source (A5):** chain-lifecycle upstream §8.6
- **Rationale (A1):** Any Snapshot — birth, cadence, or terminator — is a chain
  anchor at its Sequence, so a single recovery path re-seeds the consumer from
  the freshest anchor and bounds catch-up cost by the cadence interval.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-03` Consumer
- **Priority (A32):** P1
- **Criticality (A33):** High

#### <a id="r-cl-035"></a>`R-CL-035` — Recovery triggers and observability

> *A conforming consumer SHALL trigger anchor recovery (R-CL-034) when a selected
> threshold is exceeded (candidate families: absolute taint size, taint density,
> taint age, field-level taint impact, or a consumer SLO) and SHALL expose, at
> minimum, per-chain frontier and taint metrics (size, age, density),
> chain-recovery events, and taint signatures propagated into derived outputs.
> Concrete thresholds and metric backends are deployment selections.*

- **Type (A38):** Functional
- **Parent need (A4):** `N-CL-002` Gap-Tolerant Consumption; `N-CL-005` Bounded
  Recovery Latency
- **Source (A5):** chain-lifecycle upstream §8.7, §8.8; platform need
  [N-EDDT-009][n-eddt-009] (Platform Operational Health)
- **Rationale (A1):** Recovery is only useful if it fires before uncertainty
  becomes unacceptable; exposing taint metrics and propagating taint signatures
  lets operators and downstream analytics act on partiality deliberately.
- **V&V method (A2):** Test
- **Allocation (A8):** `S-CL-03` Consumer
- **Priority (A32):** P2
- **Criticality (A33):** Important

## 8. Design Notes *(informative)*

### 8.1 Chain lifecycle narrative

A chain is a single entity's history as a single-writer, ordered log. It begins
with a birth Snapshot at `Sequence == 0` (R-CL-021), advances through deltas and
periodic cadence Snapshots, and ends — if it ends — with a terminator
(R-CL-023). When an entity is re-identified, a symmetric reset (R-CL-022) closes
the predecessor and opens a successor, linked by `NextChainID` / `PreviousChainID`
so the graph is navigable in both directions. Producer rotation (R-CL-024) is
deliberately *not* a lifecycle event: it changes the writer, not the chain, and
is invisible to consumers. The illustrative reset triggers for a domain entity —
for a pump, say, decommissioning and replacement, a sensor swap that re-bases
calibration, or a re-provisioning that rebinds identity — are per-entity
catalogue policy, not part of this contract.

### 8.2 Consumer state machine rationale

The consumer keeps two pieces of per-chain state: a frontier (high-water mark)
and a taint set (the gaps behind it). The receive protocol (R-CL-032) is
deliberately small because it separates three concerns that are easy to
conflate: duplicates (dispatched by the frontier), gaps (applied across, with
taint recorded), and accumulated loss (carried as taint until recovered). This
yields exactly-once apply over at-least-once transport without a dedup log or a
reordering buffer in the base case.

### 8.3 Late-arrival policies

Strong rejection (R-CL-033) is the required floor: simple, correct, and it makes
every gap eventually demand a recovery. Jitter-buffered apply is a consumer-side
optimisation that heals briefly-out-of-order arrivals within a bounded window
with no producer or protocol change; it is opt-in because latency-sensitive
consumers prefer apply-on-arrival to a delayed-but-contiguous view. CRDT-style
strong handling for `delta.commutative` fields is reserved.

### 8.4 Snapshot cadence rationale

Cadence Snapshots (R-CL-027) are full-state re-anchors. The time-based floor
bounds recovery latency predictably; the change-driven trigger fires on
domain-significant transitions regardless of clock. Each component covers the
other's blind spot, which is why the production default is hybrid rather than
either alone.

### 8.5 Chain ownership and failover

Single-writer ownership (R-CL-028) is the property that makes a monotonic
Sequence achievable without consensus and lets chains shard across producers.
The mapping mechanism (R-CL-029) is a deployment choice; whatever is chosen must
preserve single-writer and monotonicity across rotation. Failover (R-CL-030)
must answer who takes over, from which Sequence (anchor recovery answers this by
reading the latest Snapshot), and how split-brain is fenced.

### 8.6 Disposal

Per-chain disposal (R-CL-026) is deletion of everything keyed by an opaque
`ChainID`, which is PII-safe by construction (R-CL-003). Field-level logical
erasure uses an `OpRetract` `FieldDelta`; regulatory physical erasure may add an
out-of-band rewrite of historical partitions. Cross-chain disposal needs an
external natural-key-to-chain index, whose shape this contract leaves to the
deployment.

### 8.7 Division of responsibility with delta-gen

The runtime enforces the chain-integrity, sequence, EffectiveAt, and provenance
invariants (R-CL-014…R-CL-018) *inside* `HeaderAfterApply` / `HeaderForDiff`;
`delta-gen`-generated `Apply` / `Diff` *call* those functions and propagate the
error (delta-gen spec R-DG-029…R-DG-032). The tri-state `FieldDelta` carrier
(R-CL-006) is a runtime type, but the per-field application is generated as an
inline switch rather than a runtime `ApplyFieldDelta` helper (E-CL-003;
delta-gen spec R-DG-016). This keeps the algebra in generated code and the
envelope contract in the runtime, with no duplicated enforcement.

### 8.8 Subject transport

Chain notifications are carried on a transport whose subjects follow the
deployment's subject-naming standard — per-tier addressability, chain-anchor
recovery request/response, and subject-filter subscription all live in that
standard. This specification is agnostic to it and assumes a sufficient
subject-naming standard is followed; it defines no subject grammar of its own.
