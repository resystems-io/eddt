# delta-gen — Code Generator Specification

This document specifies the `delta-gen` code generator: the input
grammar (annotated Snapshot Go structs), the output grammar (the
companion Delta type and its `Apply`, `Diff`, `Coalesce`, and
`EntityID` operations), the per-field emission rules, the runtime
contract surface that the generated code depends on, and the
algebraic invariants the generator must uphold.

This specification is **normative**. Conforming generator
implementations SHALL satisfy the requirements stated below. Sections
explicitly marked *(informative)* are non-normative.

A reader unfamiliar with the broader Event-Driven Digital Twin (EDDT)
should first read [eddt-context.md](eddt-context.md) for the
architectural context and [eddt-glossary.md](eddt-glossary.md) for
the canonical vocabulary. The high-level needs that this generator
contributes to are listed in [eddt-needs.md](eddt-needs.md) and
traced upward from the needs in §3 below.

**Status:** Draft v0.1 — Needs section complete; Requirements section
to follow in a subsequent commit.

## 1. Purpose and Scope

### 1.1 Purpose

The `delta-gen` generator reads a Go struct that has been annotated
to declare its delta semantics and emits the companion `Delta` type
together with the operations that constitute the chain-lifecycle
contract:

- **`Apply`** — fold a delta into a snapshot, yielding the next
  snapshot.
- **`Diff`** — compute the minimal delta that transforms one snapshot
  into another.
- **`Coalesce`** — combine two sequential deltas into a single
  effective delta without referring back to a snapshot.
- **`EntityID`** — derive a deterministic, content-hashed identifier
  for the entity a snapshot represents.

The generated code is the substrate on which Event-Driven Digital
Twin (EDDT) snapshots are propagated, replicated, persisted, and
queried.

### 1.2 In Scope

This specification covers:

- The input grammar — the Go syntactic and semantic shape an
  annotated Snapshot struct must take, including the `eddt:"…"` tag
  vocabulary.
- The output grammar — the generated `Delta` type and its companion
  functions.
- The per-field emission rules — how a field's Go type and tag
  combination determine its representation in the Delta type and its
  contribution to each operation.
- The runtime contract surface — the types and helper functions
  defined under `go.resystems.io/eddt/runtime` that the generated
  code depends on.
- The algebraic invariants — the laws (round-trip, identity-diff,
  fold-equivalence, determinism) that conforming generated code must
  satisfy.
- The CLI surface and `//go:generate` ergonomics.

### 1.3 Out of Scope

This specification does not cover:

- Runtime transport mechanics (NATS subjects, queue groups, broker
  configuration).
- Persistence layouts beyond the Arrow / Parquet round-trip
  guarantees referenced in §3.
- The content-based filtering, relation-set matching, or routing
  semantics implemented atop delta notifications by sibling
  subsystems.
- The cellular, mining, manufacturing, healthcare, or other
  application domain semantics for any specific Snapshot type — those
  are properly the concern of the domain module authoring the
  Snapshot.

## 2. Notation and Conventions

### 2.1 Conformance Language

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
"SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and
"OPTIONAL" in this document are to be interpreted as described in
RFC 2119 [[RFC2119]](#6-references) and RFC 8174
[[RFC8174]](#6-references) when, and only when, they appear in all
capitals, as shown here.

These conformance keywords bind only `R-DG-NNN` requirement
statements (§4). Need statements (§3) use the INCOSE canonical form
(§2.2) and do not carry RFC 2119 obligation force.

### 2.2 Need Statement Form

Each `N-DG-NNN` need is stated in the INCOSE Guide to Writing
Requirements canonical form
[[INCOSE-GtWR]](#6-references):

> *"[Stakeholder class] needs [what] so that [why]."*

Where:

- *[Stakeholder class]* is a noun naming the role or coined class
  that originates the need; see §3.1 for the closed register of
  roles.
- *[what]* is the capability, quality, or constraint expressed in
  domain terms.
- *[why]* is a brief rationale; the full rationale appears as the
  `Rationale (A1)` bullet of the entry.

Need statements describe expectations; they do not impose obligations
in the RFC 2119 sense. Requirements derived from a need (§4) carry
the obligation force.

### 2.3 Identifier and Reference Conventions

Identifiers in this specification follow flat namespaces with the
domain prefix `DG`:

- **`N-DG-NNN`** — needs (this document, §3).
- **`R-DG-NNN`** — requirements (this document, §4).
- **`E-DG-NNN`** — errata (this document, §5).

Identifiers do not encode hierarchy; sub-section breaks are
structural. Cross-references within this document appear inline as
e.g. `N-DG-003` and resolve via the link block at the foot of this
document, maintained by `scripts/refs-linkify.py`.

External references cite the source document by short name
(e.g. *chain-lifecycle spec*) or by formal citation in §6
References. INCOSE attribute references appear inline as `(A18)`,
`(A5)`, etc.; attribute definitions live in the
[INCOSE Guide to Writing Requirements](#6-references).

### 2.4 Code and Type Notation

Go syntactic fragments appear in fenced code blocks tagged `go`. A
small synthesised example — a pump on a manufacturing line, with a
single payload field — illustrates the minimum shape of an annotated
Snapshot:

```go
type PumpSnapshot struct {
    runtime.Header
    SerialNumber string  `eddt:"entity.key"`
    PressureKPa  float32
}
```

AST shape notation, where used informally, follows the conventions
of `go/ast`. Field-tag grammar, where formalised, is presented in
EBNF.

Where this specification refers to Go's standard library or to the
`go.resystems.io/eddt/runtime` package without further qualification,
it refers to the version current at the document's status date.

### 2.5 Normative versus Informative

Sections explicitly marked *(informative)* — and the introductory
paragraph of the document — are non-normative. They provide context,
examples, or rationale that supports but does not constrain a
conforming implementation. All other content is normative.

## 3. High-Level Needs

This section captures the high-level stakeholder needs that
`delta-gen` exists to serve. Each `N-DG-NNN` need is a top-level
expectation; the requirements in §4 derive from one or more of these
needs through the `(A4)` trace.

The need set is small and deliberately broad. Specific technical
expectations — *deterministic Apply*, *minimal Diff output*,
*reflection-free hot path*, *byte-identical regeneration*, and so on
— surface as `R-DG-NNN` requirements traced upward to one or more of
the needs below.

### 3.1 Stakeholder Roles

The following roles originate the needs in §3.2. Each role is a
bounded responsibility; the `(A18)` bullet on each need lists one or
more of these roles. Needs whose statement subject spans multiple
roles with a shared rationale use a coined class noun that abstracts
over the listed roles; classes are introduced inline at first
mention.

| Role                   | Responsibility                                                                                                                                                                      | Primary lifecycle stage |
|:-----------------------|:------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|:------------------------|
| **Snapshot author**    | Authors an annotated Go struct that declares a Snapshot type's delta semantics through field tags.                                                                                  | Design                  |
| **Schema steward**     | Maintains and evolves Snapshot types across lifecycle iterations: adds, retires, and reshapes fields whilst preserving backward compatibility for deployed producers and consumers. | Design / Sustainment    |
| **Tool integrator**    | Wires `delta-gen` into the build pipeline (typically via `//go:generate`) and depends on its output being reproducible under configuration management.                              | Build                   |
| **Twin maintainer**    | Holds a local Snapshot at runtime and folds incoming deltas via `Apply` to keep the Snapshot current.                                                                               | Runtime                 |
| **Delta publisher**    | Computes `Diff` between successive snapshots (or emits deltas directly) and publishes the resulting deltas onto a transport, identifying entities via `EntityID`.                   | Runtime                 |
| **Analytics consumer** | Reads stored deltas and snapshots from columnar (Parquet via Arrow) or relational durable storage and reconstructs or queries historical state.                                     | Analysis                |

### 3.2 Aspirational Needs

#### <a id="n-dg-001"></a>`N-DG-001` — Declarative Authoring

*A Snapshot type custodian (the Snapshot Author or Schema Steward
roles) needs to declare and evolve a Snapshot type's delta semantics
through declarative field tags, so that introducing or maturing a
domain entity requires no per-type plumbing and remains stable across
changes to the surrounding code.*

- **Stakeholder(s) (A18):** Snapshot author, Schema steward
- **Source (A5):** Engineering convention (declarative codegen
  patterns — Go's `stringer`, `mockgen`, `enumer`); platform need
  [N-EDDT-010][n-eddt-010] (System Evolvability)
- **Rationale (A1):** Manually authoring per-type `Apply`, `Diff`,
  `Coalesce`, and `EntityID` functions for every domain entity is
  repetitive, error-prone, and brittle under schema evolution. A
  declarative annotation grammar lets a single source of truth — the
  annotated Snapshot struct — drive the generation of all four
  operations consistently, and lets schema changes propagate to every
  operation through tag-vocabulary changes rather than per-call-site
  rewrites. The same annotation grammar encodes evolution patterns
  (retired fields, additive change, clearable semantics) so the
  Snapshot type can mature across deployment iterations without
  breaking deployed producers and consumers.
- **Validation:** A Snapshot type custodian can defensibly assert
  that introducing a new domain entity requires authoring one
  annotated struct plus one `//go:generate` directive, and that
  evolving an existing Snapshot — adding fields, retiring fields,
  changing field semantics — proceeds through tag-vocabulary changes
  alone, without rewriting call sites that consume the generated
  operations.
- **Priority (A32):** High
- **Criticality (A33):** Essential

#### <a id="n-dg-002"></a>`N-DG-002` — Cross-Layer Schema Harmonisation

*A cross-POV consumer of generated types (the Delta Publisher, Twin
Maintainer, or Analytics Consumer roles) needs the same canonical
Delta and Snapshot shape to serve subject, columnar, and relational
access patterns without per-POV translation, so that the model
exposed at one Point-of-View remains structurally aligned with every
other.*

- **Stakeholder(s) (A18):** Delta publisher, Twin maintainer,
  Analytics consumer
- **Source (A5):** Platform need [N-EDDT-002][n-eddt-002]
  (Orientation & Modelling — structurally aligned POVs);
  [arrow-reader-gen.md](arrow-reader-gen.md) and
  [arrow-writer-gen.md](arrow-writer-gen.md) (sibling code generators
  in this repository)
- **Rationale (A1):** The EDDT exposes its model through six
  Points-of-View — subject, columnar, and relational access patterns,
  each over snapshot and deltaflow data natures. Schema-level
  translation between POVs introduces drift, duplicated maintenance
  effort, and impedance mismatch at every boundary. By generating a
  canonical Delta and Snapshot pair from which the subject payload,
  the Arrow columnar projection (via `arrow-writer-gen` /
  `arrow-reader-gen`), and the relational table layout are all
  derived structurally, the system preserves alignment across access
  patterns by construction.
- **Validation:** A cross-POV consumer can defensibly assert that
  the generated Delta and Snapshot types serve unchanged across
  subject publication on a transport, columnar storage in Parquet via
  Arrow, and relational query, with field names, types, and ordering
  carried structurally across the access patterns.
- **Priority (A32):** High
- **Criticality (A33):** Essential

#### <a id="n-dg-003"></a>`N-DG-003` — System Traceability

*A traceability consumer (the Twin Maintainer for runtime provenance,
the Tool Integrator for reproducible codegen, or the Analytics
Consumer for historical state) needs the generated artefacts to
support audit-trail reasoning, so that any state the system produces
can be explained from its inputs and reproduced from its source.*

- **Stakeholder(s) (A18):** Twin maintainer, Tool integrator,
  Analytics consumer
- **Source (A5):** Platform need [N-EDDT-008][n-eddt-008] (Data
  Completeness & Fidelity); platform need [N-ECOL-007][n-ecol-007]
  (Configuration & Baseline Management); chain-lifecycle spec
  (`Header` carriage)
- **Rationale (A1):** Three traceability concerns share a common
  rationale: the system's outputs are explained by their inputs at
  every layer. (i) Twin maintainers must reason about the freshness,
  origin, and lineage of state they hold — generated `Apply` and
  `Diff` therefore propagate the `runtime.Header` provenance metadata
  forward through every state transition. (ii) Tool integrators must
  reproduce identical generated output across regenerations of the
  same source, so the generated baseline can be configuration-managed
  and verified in CI. (iii) Analytics consumers must reconstruct any
  past state of any entity from durable storage. The need is
  satisfied through deterministic generator output, forward-propagated
  provenance metadata in every operation, and durable columnar
  persistence of every observed state and transition.
- **Validation:** A traceability consumer can defensibly assert that
  for any state the runtime produces, the inputs that gave rise to
  it can be identified — runtime provenance is carried forward
  through `Apply` and `Diff`, generator output is byte-identical
  across repeat runs over the same source, and historical state is
  recoverable from durable Parquet storage of past deltas and
  snapshots.
- **Priority (A32):** High
- **Criticality (A33):** Essential

#### <a id="n-dg-004"></a>`N-DG-004` — Chain Safety

*A chain participant (the Twin Maintainer consuming a chain, or the
Delta Publisher producing one) needs delta-driven state movement to
remain correct under disorder, replay, and partial failure, so that
distributed twins converge on consistent state without coordinated
writes and without losing safety in the presence of transient
transport faults.*

- **Stakeholder(s) (A18):** Twin maintainer, Delta publisher
- **Source (A5):** Chain-lifecycle spec (`Chain`, `Sequence`,
  `ChainID`, `Frontier`, `Taint` semantics);
  [[MühlEtAl2006]](#6-references) (decoupling triad — space, time,
  sync; idempotence under leasing); [[Lamport1978]](#6-references)
  (causal ordering and state-machine replication)
- **Rationale (A1):** Distributed delta propagation must be safe
  under realistic transport conditions — out-of-order delivery,
  duplicate delivery, dropped deltas, replay across reconnect, and
  split-horizon catch-up. The chain abstraction provides the
  substrate for safety: a Snapshot anchored at `Sequence = 0`,
  strictly monotonic deltas, single-writer ownership of the chain's
  Sequence space, and content-hashed identification that lets
  producers and consumers route by `EntityID` without coordinated
  naming. The generated `Apply`, `Diff`, `Coalesce`, and `EntityID`
  operations must honour the chain semantics consistently — so that
  determinism gives convergence, idempotence handles re-delivery,
  content-hashed routing eliminates coordination, and `Coalesce`
  associativity lets a slow consumer fold a backlog without going
  back to the snapshot.
- **Validation:** A chain participant can defensibly assert that any
  two replicas seeded from the same initial Snapshot and fed the same
  set of deltas — in any order, with any duplicates, with any subset
  re-delivered within an acceptable reordering window — produce
  equivalent observable Snapshots, and that publishers and consumers
  can route by `EntityID` without coordinating naming across
  publishers.
- **Priority (A32):** High
- **Criticality (A33):** Essential

#### <a id="n-dg-005"></a>`N-DG-005` — Operational Performance

*A throughput-sensitive consumer of generated code (the Twin
Maintainer, Delta Publisher, or Analytics Consumer roles) needs the
delta operations and their durable representation to be bounded by
domain logic and storage characteristics rather than by framework
overhead, so that real-time event processing, ingest, and analytical
workloads all scale to the data volumes characteristic of large
instrumented infrastructure observation.*

- **Stakeholder(s) (A18):** Twin maintainer, Delta publisher,
  Analytics consumer
- **Source (A5):** Platform needs [N-EDDT-005][n-eddt-005] (Temporal
  Performance) and [N-EDDT-007][n-eddt-007] (Scalability & Data
  Velocity)
- **Rationale (A1):** Real-time signal generation, high-cardinality
  entity tracking, and analytical scan workloads each impose
  different but bounded performance envelopes. The generator must
  produce code whose hot paths are free of runtime reflection, whose
  Delta output minimises wire size against the corresponding Snapshot
  diff, and whose type shape projects efficiently into Arrow's
  columnar layout. Performance characteristics that are not bounded
  by domain logic — needless allocations, per-field reflection, fat
  envelope encodings — would prevent the system from operating at
  the data velocities and entity counts characteristic of the
  instrumented domains the platform is designed to serve.
- **Validation:** A throughput-sensitive consumer can defensibly
  assert that the generated delta operations execute without runtime
  reflection on the hot path, that the Delta wire size scales with
  the count of changed fields rather than the count of total fields,
  and that the generated types round-trip through Arrow's columnar
  layout without per-row marshalling overhead.
- **Priority (A32):** High
- **Criticality (A33):** Essential

## 5. Errata Register

The errata register records discrepancies between the
specification's intent and its realisation, supersession of entries,
and resolved defects. The register accumulates forward: resolved
errata are not removed but are marked resolved with their resolution
date.

Errata are identified `E-DG-NNN`. Each erratum records:

- The affected `N-DG-NNN` or `R-DG-NNN` identifier(s).
- The discrepancy or defect.
- The resolution (or *open*, with the date opened).
- The resolution date once closed.

At v0.1 the register is empty.
